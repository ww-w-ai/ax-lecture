# [부록] 하네스는 프롬프트를 어떻게 조립하는가 — Claude Code · Codex · Hermes · OpenClaw

> 같은 LLM이라도 **하네스(에이전트 실행기)** 가 시스템/유저 프롬프트를 어떻게 조립하느냐가 동작을 가른다.
> 4종 하네스의 **실제 소스를 직접 분석**한 결과(추론 시 요청 조립 기준, 학습 아님).
> 상태: REFERENCE (2026-06). 소스 라인번호는 버전에 따라 이동 가능.

분석 소스:
- **Claude Code** v2.1.88 (TypeScript)
- **Codex** (OpenAI, `codex-rs` Rust)
- **Hermes Agent** (Python)
- **OpenClaw** (TypeScript)

---

## 0. 먼저 — 4종에서 공통으로 확인된 5가지 (가장 중요)

1. **전부 Stateless.** 4종 모두 매 호출에 **전체 대화 히스토리를 통째로 재전송**한다. 모델이 기억하는 게 아니라 하네스가 매번 다시 조립해 보낸다. (핸드아웃 0.5 무상태성의 실증.)
2. **도구 정의(스키마)는 프롬프트 텍스트가 아니다.** 4종 모두 도구·MCP 스키마를 **별도 `tools` 필드(배열)** 로 보낸다. 시스템 프롬프트엔 "도구 사용 *안내*"만 산문으로. (흔한 오해 정정.)
3. **안정 → 동적 순서 + 캐시 경계.** 4종 모두 불변(정체성·규칙)을 앞, 변동(메모리·날짜·런타임)을 뒤에 두고 **캐시 경계**로 가른다. 앞부분을 프롬프트 캐시에 태워 비용·지연을 줄이려는 동일 전략.
4. **프로젝트 규칙 파일(CLAUDE.md / AGENTS.md) 위치는 갈린다** (반직관 포인트):
   - Claude Code·Codex → **유저 메시지**로 주입.
   - Hermes·OpenClaw → **시스템 프롬프트 텍스트**에 인라인.
5. **역할(role) 관례도 제각각.** 특히 **Codex엔 `system` 역할이 아예 없다**(최상위 `instructions` 필드 사용). 나머지는 system을 쓰되 형태가 다름.

> 한 줄: "프롬프트 = 시스템 한 덩어리"가 아니다. **(a) 시스템 + (b) 별도 tools 배열 + (c) 유저/대화 배열** 세 표면으로 나뉘고, 무엇을 어디에 넣느냐가 하네스마다 다르다.

## 1. 한눈 비교표

| 항목 | Claude Code | Codex | Hermes | OpenClaw |
|---|---|---|---|---|
| API/와이어 | Anthropic Messages (`system` 파라미터 분리) | **Responses API** (`instructions` 필드) | OpenAI chat-completions 내부표준 → 프로바이더별 변환 | Anthropic Messages (`system` 파라미터) |
| 역할 집합 | system / user / assistant / tool | **system 없음** — developer / user / assistant | system / user / assistant / tool | user / assistant / tool_result (+ system 파라미터) |
| 시스템 빌더 | `getSystemPrompt()` 문자열 배열 | 모델별 `base_instructions` 문자열 | `build_system_prompt()` 3계층 | `buildAgentSystemPrompt()` 문자열 |
| 도구 스키마 | top-level `tools` | 별도 `tools` 배열 | 별도 `tools` 필드 | 별도 `tools` 파라미터 |
| 규칙파일 위치 | **user**(`<system-reminder>`) | **user** 메시지 | **system** 인라인 | **system** 인라인 |
| 메모리 | auto-memory→**system** | 초기 컨텍스트 시드 | 휘발 계층(system 하단) | system 섹션 |

| 히스토리 | 매 호출 전체 재전송 | 매 턴 전체 input 재구성 | 매 턴 fresh agent + 전체 재생 | 매 호출 전체 재전송 |

## 2. 공통 골격 (5단면으로 보기)

어떤 하네스든 매 호출은 결국 세 표면으로 나간다:
- **(A) 시스템 프롬프트** — 정체성·규칙·도구 사용 안내·환경. (캐시 안정 영역)
- **(B) `tools` 배열** — 도구·MCP 스키마(JSON). 프롬프트 텍스트 아님.
- **(C) 입력/대화 배열** — 유저 메시지 + 누적 대화(+ 규칙파일·메모리·날짜가 여기 끼기도).

차이는 "무엇을 A에, 무엇을 C에 넣느냐"와 "캐시 경계를 어디 긋느냐"다.

## 3. 하네스별 요점 (소스 근거)

### Claude Code (v2.1.88)
- 시스템: `constants/prompts.ts:444 getSystemPrompt()` → 정체성(`"You are Claude Code…"`) → 도구사용안내 → **경계 마커**(`:572`) → 동적(auto-memory·env·MCP instructions).
- **프로젝트 CLAUDE.md는 시스템이 아니라 유저 메시지**(`utils/api.ts:449 prependUserContext` → `<system-reminder># claudeMd … # currentDate`). ← 반직관 핵심.
- 도구: `utils/api.ts:169 {name, description, input_schema}`. MCP desc는 2048자 컷(`mcp/client.ts:218`), inputSchema는 무삭제.
- 캐시: 시스템 텍스트 블록 scope(prefix/global) + 메시지 마지막 1곳 마킹(`claude.ts:3063 addCacheBreakpoints`).

### Codex (codex-rs)
- **Responses API**, `system` 역할 없음 — 최상위 `instructions`(모델별 `base_instructions`) + `input` 배열 + `tools` 배열 3분리 (`client.rs:791`).
- 환경·권한·스킬·앱안내 = **developer 메시지**, **AGENTS.md·환경컨텍스트 = user 메시지**로 묶어 시드 (`session/mod.rs:2746 build_initial_context`).
- AGENTS.md = user-role, prefix `"# AGENTS.md instructions for"` (`user_instructions.rs`).
- 매 턴 로컬 히스토리에서 input 전체 재구성(`store:false`, stateless).

### Hermes Agent (Python)
- 시스템 3계층 `"\n\n"` 결합, **안정→컨텍스트→휘발** (`system_prompt.py:336`): 정체성(SOUL.md) → 도구가이드 → 스킬인덱스 → (컨텍스트파일 인라인) → 메모리·**날짜만 타임스탬프**(`:305`, 캐시 유지용).
- **규칙파일(AGENTS.md/CLAUDE.md/.cursorrules) = 시스템 텍스트에 인라인**(first-match, 20k자 컷).
- 도구 = 별도 `tools` 필드, 회상/플러그인 컨텍스트는 **유저 메시지**에 합침(시스템 prefix 바이트 고정).
- 멀티 프로바이더 transport(anthropic/bedrock/codex 변환). 서버리스·크론형 → 매 턴 fresh agent + 세션DB 캐시 프롬프트 복원.

### OpenClaw (TypeScript)
- 시스템 `buildAgentSystemPrompt()`(`system-prompt.ts:425`): 정체성 → 도구안내 → 안전규칙 → 스킬("SKILL.md 읽어라" 포인터) → 메모리 → **프로젝트 컨텍스트(AGENTS.md verbatim)** → **`<!-- OPENCLAW_CACHE_BOUNDARY -->`** → 동적(런타임 정보).
- `CLAUDE.md`는 `AGENTS.md`의 **심볼릭 링크**(파일 단일화).
- 도구/MCP = 별도 `tools` 배열(네이티브·MCP 무구분 머지). 캐시: 경계 위 ephemeral `cache_control` + 마지막 user 블록도 마킹(누적분 캐시), TTL 호스트별 1h/짧게.
- OAuth(구독키) 경로면 정체성을 `"You are Claude Code…"`로 덮고 도구명을 Claude Code 표준명으로 매핑.

## 4. 빌더(우리)에게 주는 교훈

- **상세 가이드·예시는 `tools`의 inputSchema(안 잘림)나 유저 메시지로** — 시스템/툴 description은 컷(예: Claude Code 2048자)될 수 있다.
- **불변을 앞, 변동을 뒤** + 캐시 경계 의식 → 프롬프트 캐시 적중률↑ = 비용·지연↓.
- **규칙파일을 system에 둘지 user에 둘지**는 트레이드오프(캐시 안정 vs 우선순위·가시성). 하네스마다 답이 다르다.
- **무상태성**을 전제로 설계: 긴 작업은 요약·박제로 컨텍스트를 가볍게, 세션 경계를 의식.

## 부록. 결국 도달하는 곳 — LLM API 규격 (신형 API 기준)

하네스가 시스템/유저/도구를 분리해 조립하는 이유는, **결국 그 모양으로 API를 호출**하기 때문이다. 두 신형 API를 비교한다. (Claude = 공식 레퍼런스 확인 / OpenAI = 공개 Responses API 기반, 지식 컷오프 2026-01)

> **멘탈 모델 (이거 하나면 됨)**
> - **OpenAI Chat Completions 규격 = 사실상 업계 공용어.** DeepSeek·MiniMax·GLM·MiMo 등 신형(특히 중국) 모델 대부분이 이 규격을 그대로 따른다 → `model` 슬러그만 바꾸면 같은 코드 재사용.
> - **"호환"은 공통분모(messages·tools·response_format)까지.** 벤더 고유 기능(thinking/reasoning, GLM `clear_thinking`, DeepSeek thinking 모드 등)만 모델별 추가 필드로 얹는다.
> - **OpenRouter는 선택(편의 게이트웨이).** 의무 경로가 아니라 "여러 모델을 한 키·한 포맷으로" 쓰고 그 reasoning 차이를 정규화하고 싶을 때의 옵션. 단일 모델 프로덕션·비용 최적화는 네이티브 직접 호출이 유리.
> - **진짜 다른 규격은 Anthropic(Claude) 하나뿐** — system 별도 필드, tool_result 구조가 OpenAI와 다름(아래 A vs B). 그래서 이 부록이 Claude / OpenAI / OpenRouter 셋으로 나뉜다.

### A. Claude — Messages API (`POST /v1/messages`)
```jsonc
{
  "model": "claude-opus-4-8",
  "max_tokens": 16000,                 // 하드 상한(모델은 모름)
  "system": [                          // 별도 최상위 필드 (역할 배열 밖)
    { "type": "text", "text": "정체성·규칙·페르소나",
      "cache_control": { "type": "ephemeral", "ttl": "1h" } } ],   // 캐시 경계
  "messages": [
    { "role": "user", "content": "질문" },
    { "role": "assistant", "content": [ {"type":"tool_use","id":"...","name":"...","input":{}} ] },
    { "role": "user", "content": [ {"type":"tool_result","tool_use_id":"...","content":"..."} ] }
  ],
  "thinking": { "type": "adaptive", "display": "summarized" },  // 신모델은 adaptive만
  "output_config": {
    "effort": "high",                  // low|medium|high|xhigh|max
    "format": { "type":"json_schema", "schema": {...} },   // 구조적 출력
    "task_budget": { "type":"tokens", "total": 64000 } },  // [beta] 에이전트 토큰예산
  "tools": [
    { "name":"get_weather", "description":"...", "input_schema":{...}, "strict": true },
    { "type":"web_search_20260209", "name":"web_search" },        // 서버 도구: 웹검색
    { "type":"web_fetch_20260209", "name":"web_fetch" },          // URL 본문
    { "type":"code_execution_20260120", "name":"code_execution" },// 코드 실행
    { "type":"memory_20250818", "name":"memory" } ],
  "tool_choice": { "type":"auto", "disable_parallel_tool_use": false },
  "mcp_servers": [ {"type":"url","name":"...","url":"..."} ],     // [beta] 원격 MCP
  "context_management": { "edits": [ {"type":"clear_tool_uses_20250919"},
                                     {"type":"compact_20260112"} ] }, // [beta]
  "cache_control": { "type":"ephemeral" },
  "stream": true,
  "speed": "fast",                     // [beta] 최대 2.5배
  "stop_sequences": ["..."]
  // ⚠️ 신모델은 temperature·top_p·top_k 제거 → 보내면 400
}
```
- 헤더: `x-api-key`, `anthropic-version`, `anthropic-beta`. 토큰 세기 = `/v1/messages/count_tokens`.

### B. OpenAI — Responses API (`POST /v1/responses`)
```jsonc
{
  "model": "...",
  "instructions": "시스템 역할 (별도 필드 — system role 없음)",
  "input": [
    { "role":"developer", "content":"규칙" },
    { "role":"user", "content":"질문" },
    { "type":"function_call_output", "call_id":"...", "output":"도구 결과" } ],
  "max_output_tokens": 16000,
  "reasoning": { "effort":"high", "summary":"auto" },   // Claude의 thinking에 해당
  "text": { "format": { "type":"json_schema", "name":"...", "strict": true, "schema": {...} } },
  "tools": [
    { "type":"function", "name":"...", "parameters":{...}, "strict": true },
    { "type":"web_search" },                              // 호스티드: 웹검색
    { "type":"file_search", "vector_store_ids":[...] },   // 문서검색(RAG)
    { "type":"code_interpreter", "container":{"type":"auto"} },
    { "type":"mcp", "server_label":"...", "server_url":"..." } ],
  "tool_choice": "auto",                                  // auto|none|required|{type,name}
  "parallel_tool_calls": true,
  "stream": true,
  "store": true,                       // 서버에 응답 저장(상태 보관 옵션)
  "previous_response_id": "resp_...",  // 이어가기 — 서버 stateful
  "include": ["web_search_call.results"],
  "temperature": 1.0, "top_p": 1.0     // 비추론 모델만
}
```
- 헤더: `Authorization: Bearer <key>`.

### C. OpenRouter — 통합 게이트웨이 (`POST /api/v1/chat/completions`)
수백 개 모델을 **단일 OpenAI Chat Completions 호환** 엔드포인트로 감싸고, 벤더 차이를 흡수하는 확장 필드를 얹는다. base_url만 바꿔 OpenAI SDK 그대로 사용.
```jsonc
{
  // ── OpenAI 호환 ──
  "model": "deepseek/deepseek-v4-pro",   // 슬러그 org/model
  "messages": [ { "role":"user", "content":"..." } ],
  "temperature": 1.0, "top_p": 1.0, "max_tokens": 4096, "seed": 42, "stop": ["..."],
  "stream": true,
  "response_format": { "type":"json_schema", "json_schema": {"name":"...","strict":true,"schema":{}} },
  "tools": [ {"type":"function","function":{"name":"...","parameters":{}}} ],
  "tool_choice": "auto",

  // ── 확장: 비OpenAI 샘플링(지원 프로바이더에만 전달, 나머진 무음 무시) ──
  "top_k": 0, "repetition_penalty": 1.0, "min_p": 0.0, "top_a": 0.0,

  // ── 확장: reasoning 통합 제어(벤더별 thinking 차이를 하나로 흡수) ──
  "reasoning": { "effort":"high",        // max|xhigh|high|medium|low|minimal|none
                 "max_tokens": 8000, "enabled": true, "exclude": false },

  // ── 확장: 라우팅·폴백 ──
  "models": [ "deepseek/deepseek-v4-pro", "z-ai/glm-5.2" ],  // 1차 실패 시 순차 폴백
  "route": "fallback",
  "transforms": ["middle-out"],          // 컨텍스트 초과 시 압축

  // ── 확장: 프로바이더 선택 ──
  "provider": { "order":["..."], "allow_fallbacks": true, "require_parameters": true,
                "data_collection":"deny", "only":["..."], "ignore":["..."],
                "quantizations":["fp8"], "sort":"throughput" },

  // ── 확장: 사용량 회계 ──
  "usage": { "include": true }
}
```
- 핵심 가치: **벤더별 thinking·도구 차이를 OpenAI 스키마 + `reasoning`/`provider` 확장으로 단일화.** 모델 교체 = 슬러그 한 줄.

### 최근 OpenRouter 상위 모델 (이 게이트웨이가 감싸는 신형들)
| 모델 | 슬러그 | 컨텍스트 | Reasoning | Tools | 비고 |
|---|---|---|---|---|---|
| DeepSeek V4 | `deepseek/deepseek-v4-pro` | 1M | `thinking:{type}` + `reasoning_effort`, `reasoning_content` 분리 | 예 | preview(2026-04), 가격 추정 |
| MiMo 2.5 (Xiaomi) | `xiaomi/mimo-v2.5`(+`-pro`) | 1M | API 노출 방식 **미확인** | 추정 | omnimodal·agentic |
| MiniMax M3 | `minimax/minimax-m3` | 1M | `thinking`(기본 ON) + `reasoning_split` | 추정 | OpenAI 호환 |
| GLM 5.2 (Z.ai) | `z-ai/glm-5.2` | 1M | `thinking:{type, clear_thinking}` + `reasoning_effort` | 추정 | 벤더필드는 `extra_body` |

> 패턴: **신형 모델들이 `thinking` 객체 + `reasoning_content` 분리 필드로 수렴**, 컨텍스트 1M이 표준, 전부 OpenAI Chat Completions 호환 → OpenRouter가 그 차이를 `reasoning`/`provider` 확장으로 흡수.

### 핵심 대조 (사고·검색·상태 포함)
| 옵션 | Claude Messages | OpenAI Responses |
|---|---|---|
| 시스템 위치 | 최상위 `system` 필드 | 최상위 `instructions` 필드 |
| 사고/추론 | `thinking:{type:"adaptive"}` + `output_config.effort` | `reasoning:{effort, summary}` |
| 사고 노출 | `thinking.display` | `reasoning.summary` |
| 웹 검색 | `tools:[{type:"web_search_20260209"}]` | `tools:[{type:"web_search"}]` |
| 문서검색(RAG) | 직접 임베딩/MCP | `tools:[{type:"file_search"}]` |
