# 슬라이드 구성안 — 홀수(이미지)/짝수(설명) 페어 방식

> AI 시대의 바이브코딩과 에이전트 활용법 (슬라이드 구성안)
> 상태: ACTIVE-PLAN (2026-06-27)

## 포맷 규칙
- **홀수 페이지 = 이미지만 (스티브 잡스식)**: 사진 1장, **텍스트·문구 없음**. 청중이 "이 이미지가 뭘 뜻할까?" 맞춰보게 하고, 강사가 말로 푼다. (표지 P1 제외 — 타이틀 포함)
- **짝수 페이지 = 설명 (읽기 자료)**: 바로 앞 홀수 이미지의 의미를 글로 풀어쓴 본문. 슬라이드를 *읽어서* 학습 가능 (교과/자습용). **이미지의 텍스트는 전부 여기에 있다.**
- 흐름: P1(이미지) → P2(P1 설명) → P3(이미지) → P4(P3 설명) …
- 발표 시: 홀수(사진)만 띄우고 → 청중 추측 유도 → 강사 구두 설명 / 배포·자습 시: 홀수+짝수 같이 읽음.

## [오프닝] AI 시대, 관점의 전환

### P1 [이미지] — 표지
- 비주얼: 텅 빈 도로가 지평선으로 뻗는 사진 + 큰 타이틀.

### P2 [설명] — 오늘의 약속
- 청중은 이미 Claude Code·Max 사용자. 그래서 "AI가 대단하다"는 설교는 안 함.
- 오늘은 세 개의 **비자명한 통찰** + 세 개의 레버(만들기·자동화·자율주행).
- 큰 질문: "코드는 짤 줄 아는데, 왜 아직 내 일은 혼자 굴러가지 않는가?"

### P3 [이미지] — 관점① 시대의 이름
- 비주얼: **인터넷 / 모바일 / AI** 3개 이미지를 화면 꽉 채워 3분할 병치 (각 시대를 상징하는 풀블리드 사진).

### P4 [설명] — AI 위에 사업을 올려라
- 아직도 많은 사람이 AI를 *방문해서 한 번씩 쓰는 서비스*로 본다.
- 인터넷·모바일이 그랬듯, 진짜 변화는 AI가 **모든 일의 바닥에 깔린 인프라**가 된 것.
- 사고의 전환: AI를 *쓴다* → **AI 인프라 *위에* 내 서비스를 올린다**.
- "AI 기능을 붙인 앱"과 "AI 인프라 위에 설계한 사업"은 완전히 다른 사고다.

### P5 [이미지] — 관점② 새 생활권
- 비주얼: **걷기 / 자전거 / 자동차 / 비행기** 4개 이미지를 화면 꽉 채워 4분할 병치 (이동수단 세대교체).

### P6 [설명] — 새 생활권 = 새 사업 기회
- 사람들은 AI를 *놀라워는* 하지만, 그것이 **완전히 새 시대의 시작**인 줄은 모른다.
- 속도 10~50배는 *증상*일 뿐. 핵심은 **생활권의 이주**.
- 자동차가 등장하자 교외·고속도로·물류·주유소라는 **이전엔 없던 사업**이 생겼다.
- 핵심 질문: **"걷기 시대엔 불가능했지만, 자동차 시대엔 되는 사업은?"**
- (강사 메모: AI판 신사업 예시 1~2개를 여기 박아 청중의 "그래서 뭘?"에 답.)

### P7 [이미지] — 관점③ 자율주행
- 비주얼: **테슬라 이미지** 풀블리드 (자율주행 중인 모습 — 핸들에서 손 뗀 운전자 또는 오토파일럿 화면).

### P8 [설명] — 자율주행은 '내 일하는 방식'이다
- 자율주행은 *사업 아이템*이 아니라 **나의 작업 방식에 대한 지시**.
- 최근 AI가 goal을 향해 스스로 달리는 모드가 대유행 → 테슬라 오토파일럿 비유.
- 매 순간 핸들 잡기(프롬프트 직접 지시) → **목적지만 주면 엉덩이를 떼도 돌아가는** 모드.
- 혼자·소규모로 일할수록 절실: **내가 자리를 비워도 일이 진행돼야** 한다.

### P9 [이미지] — 오늘의 3 레버
- 비주얼: 세 개의 큰 레버/스위치 아이콘 (build / automate / autopilot).

### P10 [설명] — 관점 전환 = Harness 설계
- Prompt(핸들 직접) → Context → **Harness(자율주행 환경) 설계**로 관점을 올린다.
- 오늘의 3 레버: **① 만들기 = bkit / ② 자동화 = 에이전트 수지 / ③ 자율주행 = Sprint**.
- 한 줄: "인프라 위에서, 혼자 팀처럼."

## [기초] LLM과 Claude Code의 원리 — AI를 더 잘 알기

> 목적: 도구를 "왜 그렇게 써야 하는지" 알도록 원리를 *개념 수준*으로. 너무 깊지 않게, 그러나 감은 잡히게.

### P11 [이미지] — 한 토큰씩
- 비주얼: 글자/토큰이 한 조각씩 떨어져 이어지는 모습 (도미노·구슬 꿰기).

### P12 [설명] — LLM은 '다음 한 토큰'을 예측한다
- 본질: 지금까지의 텍스트 뒤에 올 **다음 토큰 하나**를 확률로 예측해 내뱉는다.
- 그걸 반복해 이어붙이면 문장·코드·논리·맥락이 된다 — 이게 놀라운 지점.
- 즉 "생각을 꺼내는" 게 아니라 **가장 그럴듯한 다음 조각**을 계속 잇는 것.

### P13 [이미지] — 토큰 하나가 만들어지는 6단계 (도식)
- 비주얼: 좌→우 파이프라인 플로우 다이어그램(6개 스테이지 박스 + 화살표, 마지막에서 1번으로 되돌아가는 루프 화살표). 도식 자체가 이미지:
```
1 입력·토큰화 → 2 임베딩+위치 → 3 트랜스포머 블록 ×N(어텐션+FFN)
  → 4 로짓(사전 점수) → 5 확률·샘플링 → 6 반복(붙이고 처음부터) ↺
```

### P14 [설명] — 6단계 상세
1. **입력 & 토큰화**: 매번 지금까지의 전체 텍스트(시스템+유저+대화)를 받는다. 목표는 단 하나 — 다음 토큰 1개. 텍스트를 자주 나오는 글자 덩어리(BPE)로 쪼개 정수 ID로.
2. **임베딩 + 위치**: 각 토큰 ID를 고차원 '의미 벡터'로. 순서를 알도록 위치 정보를 더함(없으면 "개가 사람을 물었다"와 어순 구분 불가). → `[토큰수 × 차원]` 행렬.
3. **트랜스포머 블록 ×N (핵심)**: 수십~수백 층을 통과. 각 층 = **어텐션**(각 토큰이 다른 모든 토큰을 둘러보며 "누가 중요한가" 가중치 = 맥락 이해; QKV·멀티헤드) + **FFN**(모은 정보 가공·강화, 학습된 지식 저장). 잔차·정규화로 안정. 뒤 레이어로 갈수록 표면 패턴 → 추상 의미·의도.
4. **로짓**: 모든 층 통과 후 마지막 토큰 위치의 벡터 하나가 "다음"을 압축. 이를 출력 투영에 통과 → 사전 전체(수만 토큰)에 점수.
5. **확률 & 샘플링**: 소프트맥스로 점수 → 확률(합=1). 여기서 **앞 맥락(페르소나·도메인)이 분포 자체를 바꾼다**("당신은 변호사" → 법률 토큰 확률↑). temperature·top-p로 하나를 뽑음(1등만은 아님 → 매번 조금씩 달라짐).
6. **반복 (autoregressive)**: 뽑은 토큰을 입력 끝에 붙이고 1번부터 다시 → EOS(멈춤 토큰)까지. 그 결과가 문장·코드·맥락.
- (숨은 두 장치) **씽킹**: 답 전에 추론 토큰을 비공개로 먼저 생성(학습 때부터 포함). **KV 캐시**: 3번 어텐션의 Key/Value를 저장·재사용 → 다음 토큰이 빨라지고 싸짐.

### P15 [이미지] — 채팅의 진짜 모습 (특수 토큰)
- 비주얼: 실제 와이어 포맷을 스타일된 코드블록으로 화면 꽉 채워 제시 (캡션 없이 이 블록 자체가 이미지):
```
<|system|>     ...규칙 · 페르소나...   <|end|>
<|user|>       ...질문...             <|end|>
<|assistant|>  ...답변...             <|end|>
```

### P16 [설명] — '다음 토큰 하나'로 모든 걸 커버한다
- 한 턴 안에서 똑같은 루프(다음 토큰 예측)만 반복되는데도, 아래가 전부 *저절로* 생긴다:
  - **맥락 유지** ← 매 토큰이 자기가 쓴 것을 다시 읽는다(되먹임).
  - **문장 완결** ← 완결 패턴의 확률(학습됨) — `}`·마침표가 올 확률이 높아짐.
  - **종료** ← '끝'도 하나의 토큰(EOS / `<|end|>`)을 예측하는 것.
  - **턴 구분** ← 역할 토큰(`<|user|>`·`<|assistant|>`)으로 학습된 채팅 템플릿.
- 어디에도 "문장 끝내라 / 이제 멈춰라"라는 if문이 없다 — 전부 **확률 분포에 녹아든 창발(emergence)**.
- 한 줄: **단 하나의 능력 "다음 토큰 예측"이 맥락·완결·종료·턴 구분을 다 만든다.**

### P17 [이미지] — 토큰의 종류
- 비주얼: 4색 토큰 더미(input/output/cache/thinking)에 각기 다른 가격표.

### P18 [설명] — 토큰은 왜 종류별로 가격이 다른가
- 입력(input)·출력(output)·캐시(cache)·사고(thinking) 토큰이 구분된다.
- 출력이 입력보다 비싸다(생성이 비쌈). 캐시는 재사용이라 싸다(5분/1시간 TTL).
- 활용: 반복되는 앞부분을 캐시에 태우면 비용↓. **토큰을 '돈'으로 보고 설계.**

### P19 [이미지] — 씽킹 모드
- 비주얼: 머릿속 사고 풍선은 화면 밖(비공개)에서 돌고, 최종 답만 나오는 그림.

### P20 [설명] — '생각'도 학습된 것이다
- 씽킹 = 답하기 전 모델이 펼치는 추론 과정(별도 토큰).
- 핵심: 사후 트릭이 아니라 **학습 단계부터 thinking이 포함되어 훈련**됨 → 어려운 문제 정확도↑.
- 비용·시간이 더 들지만 복잡추론엔 값어치. (그래서 설계엔 thinking ON.)

### P21 [이미지] — 페르소나/관점 주입
- 비주얼: 같은 질문에 '의사·변호사·마케터' 모자를 씌우면 답이 달라지는 그림.

### P22 [설명] — 왜 관점·도메인·페르소나를 주면 결과가 달라지나
- next-token 예측은 **주어진 맥락(조건)**에 따라 확률 분포가 달라진다.
- "당신은 20년차 보안 전문가" 같은 주입 = 그 분포를 그 영역으로 **조건화**.
- 도메인 지식·예시·관점을 넣을수록 그 방향 토큰이 뽑힐 확률↑ → 결과가 바뀐다.
- 이것이 프롬프트·컨텍스트 엔지니어링이 통하는 **근본 이유**.

### P23 [이미지] — 모든 게 프롬프트로 합쳐진다
- 비주얼: 에이전트·스킬·플러그인·CLAUDE.md 조각들이 하나의 긴 프롬프트 텍스트로 합쳐져 모델에 들어가는 그림.

### P24 [설명] — 매 호출에 무엇이 들어가나
- **시스템 프롬프트**: 정체성·규칙·도구 설명·페르소나 (항상 맨 앞, 매 호출 주입).
- **유저 프롬프트**: 그때의 요청.
- **에이전트/스킬/플러그인/CLAUDE.md**의 내용도 결국 이 프롬프트 안에 *텍스트로 합쳐져* 들어간다 — 마법이 아니라 **맥락 주입**.

### P25 [이미지] — 보이지 않는 중간 과정
- 비주얼: 채팅창(보이는 것)은 빙산 위, 수면 아래 tool call·중간 결과 더미.

### P26 [설명] — 채팅엔 안 보이는 것들
- 최종 답 외에 **tool calling(도구 호출)·도구 결과·중간 추론**이 오간다.
- 채팅 UI는 빙산의 일각, 실제론 여러 왕복이 수면 아래서 일어난다.
- 이걸 알아야 "왜 느린지/비싼지/실패했는지"가 보인다.

### P27 [이미지] — 컨텍스트 윈도우
- 비주얼: 점점 차오르는 창문/그릇에 메시지가 쌓이는 모습.

### P28 [설명] — 왜 갈수록 무거워지나 (입력 한꺼번에 / 출력 한 토큰씩)
- **입력은 한 번에, 출력은 한 토큰씩**:
  - 입력(예: 100만 토큰)은 한 방에 병렬로 훑는다 = **prefill** (첫 답까지 시간이 걸리는 이유).
  - 그 뒤 출력은 1토큰씩 = **decode** (앞서 계산한 KV 캐시 재사용).
- **길수록 제곱으로 비싸진다**: 어텐션은 토큰끼리 *서로 다 비교* → 토큰 2배면 비교량 약 4배(quadratic). 첫 응답도, 토큰당 생성 속도도 같이 느려짐.
- **매 호출마다 그 전체를 다시 보낸다**(메모리에 올라간 대화 전체) → 길수록 더 느리고 더 비쌈.
- 대응: 캐시 재사용 + **요약·박제·세션 분리**로 컨텍스트를 가볍게. (Part 3 자율주행의 핵심 기술.)
- 한 줄: **"한 번에 입력(prefill) · 한 토큰씩 출력(decode) · 길수록 제곱 비용 · 매 턴 재전송."**

### P29 [이미지] — 트랜스크립트
- 비주얼: 대화 전체가 한 줄씩 기록된 로그 파일(.jsonl) 아이콘.

### P30 [설명] — 모든 게 기록된다 → 리뷰가 된다
- 주고받은 프롬프트·tool call·토큰 수·**API 호출·rate limit 정보**가 트랜스크립트(.jsonl)에 저장.
- 이걸 분석하면 "무엇을, 얼마에, 어떻게" 했는지 **상세 리뷰**가 가능.
- 토큰 사용·한도·비용을 눈으로 보고 최적화할 수 있다.

### P31 [이미지] — claude-code-token-saver

### P32 [설명] — 우리가 만들며 알게 된 노하우

## [Part 1] bkit — 아이디어를 *안정적으로* 앱으로

### P33 [이미지] — 바닐라의 한계
- 비주얼: 같은 프롬프트, 매번 다른 결과물 3개가 들쭉날쭉(품질 그래프 톱니).

### P34 [설명] — 왜 바닐라 Claude Code로는 부족한가
- 맥락 소실: 길어지면 앞 결정을 잊는다.
- 일관성 없음: 같은 요청도 결과 품질이 매번 다르다.
- "한 번에 다 시키기"가 실패하는 이유 → 구조(공정)가 없기 때문.

### P35 [이미지] — PDCA를 코딩에
- 비주얼: 제조업 PDCA 사이클(Plan-Do-Check-Act) 화살표 링 + 코드 아이콘.

### P36 [설명] — bkit = 공정으로서의 바이브코딩
- Plan/Design(메인, thinking) → Do(구현) → Check(검증)의 사이클.
- "감으로 시키기"가 아니라 **반복 가능한 공정**으로 품질을 고정.
- 혼자라면 PM·개발·QA를 다 못 하니, **공정이 그 역할을 대신**한다.

### P37 [이미지] — 프레임워크 레벨
- 비주얼: 3단 계단 (starter → dynamic → enterprise), 개인·소규모는 앞 두 칸에 형광.

### P38 [설명] — 오버엔지니어링 피하기
- starter(간단) / dynamic(중간) / enterprise(대규모) — 단계별 복잡도.
- **개인·소규모는 starter~dynamic이 현실적.** 처음부터 엔터프라이즈 구조는 독.
- 핵심: 필요한 만큼만. 나중에 올리면 된다.

### P39 [이미지] — 이중검증
- 비주얼: 30% → 89% 품질 막대 그래프, 가운데 "검증 게이트" 통과 도식.

### P40 [설명] — 무인일수록 검증이 핵심
- 결과를 한 번 더 검증하는 게이트를 붙이면 품질이 급등.
- 사람이 일일이 못 보는 환경일수록 **자동 검증**이 안전벨트.

### P41 [이미지] — 모델 라우팅
- 비주얼: Opus(설계)·Sonnet(구현)·Haiku(잡일) 3개 엔진 + 비용/속도 다이얼.

### P42 [설명] — Claude Max 한도를 길게 쓰는 법
- Opus=설계·복잡추론 / Sonnet=구현·반복 / Haiku=단순작업.
- 모든 걸 최고 모델로 하면 한도·비용 낭비. **일에 맞는 모델 배치**.
- (실전 팁: 설계는 메인 세션 thinking, 구현은 서브로 위임.)

### P43 [이미지] — 🔧 실습 1
- 비주얼: 터미널 + `/pdca-wf` 한 줄 클로즈업.

### P44 [설명] — 실습 1 가이드
- `/plugin install bkit` (설치돼 있으면 확인) → `/pdca-wf`로 아주 작은 기능 하나.
- 예: 단일 페이지 투두 / 랜딩 섹션. **결과물보다 루프의 감각**.
- 막히면 치트시트 참고. (5분 작업, 동시 부하 고려해 작게.)

## [Part 2] 에이전트 수지 — 나만의 백오피스

### P45 [이미지] — 개발만 빨라진 함정
- 비주얼: 빠른 스포츠카(개발) 뒤에 산더미 서류(일정·메일·운영)가 발목 잡는 그림.

### P46 [설명] — 진짜 병목은 운영
- 코딩이 빨라져도 일정·메일·문서·리서치·반복 운영이 발목.
- 혼자·소규모는 백오피스 인력이 없다 → **AI에게 운영을 위임**해야.

### P47 [이미지] — MCP = 실세계 단자
- 비주얼: Claude Code에 USB-C처럼 꽂히는 여러 단자(달력·메일·웹·문서).

### P48 [설명] — MCP와 에이전트 수지
- MCP = AI를 바깥 세계(앱·데이터)에 잇는 표준 단자(USB-C 비유).
- **에이전트 수지** = Claude Code에 MCP로 붙는 Hermes형 자동화 운영 비서.

### P49 [이미지] — 수지가 하는 일
- 비주얼: 중앙 수지, 주위에 5개 아이콘(라이브러리·일정/D-day·메일·리서치·자동화 루틴).

### P50 [설명] — 수지가 주는 것
- 지식 라이브러리(문서 저장·검색·그래프).
- 시간 관리(일정·D-day·할일·자동화 루틴).
- 메일·실시간 리서치 앱(시장조사).
- 반복 운영 위임(Hermes형 자동화).

### P51 [이미지] — 아침 루틴 시나리오
- 비주얼: 커피 한 잔 + 화면에 "리서치 → 일정 정리 → 메일 초안" 자동 완료 체크.

### P52 [설명] — 시나리오: 어느 아침
- "오늘 시장 리서치 + 일정 정리 + 메일 초안"을 한 번에 위임.
- 개발 너머 **운영까지 AI가** → 혼자서도 일이 돈다.

### P53 [이미지] — 🔧 실습 2
- 비주얼: `sooji_start` 호출 화면.

### P54 [설명] — 실습 2 가이드
- 수지 MCP 연결 확인 → `sooji_start`로 기능 파악.
- 작은 자동화 1개: 일정 등록 / 문서 저장·검색 / 리서치 앱 중 택1.
- 목표: "코드 너머 운영까지" 체감.

## [Part 3] Sprint — 수 시간 자율주행 바이브코딩

### P55 [이미지] — 한 기능 vs 연쇄
- 비주얼: 좌(bkit=블록 1개) vs 우(sprint=블록들이 컨베이어로 줄줄이).

### P56 [설명] — Sprint란
- bkit이 *한 기능*이면, Sprint는 **여러 기능을 연쇄** 실행.
- 계획 → 실행 → QA → intent-audit → 배포의 전 사이클을 자율로.

### P57 [이미지] — 자율주행 3대 조건
- 비주얼: 자율주행차 3요소 — 목적지 입력(박제)·센서(검증)·긴급정지(게이트).

### P58 [설명] — 무인 주행이 성립하는 조건
1. **박제**: 명확한 design doc 단일 입력 (모호하면 자율주행 불가).
2. **검증 루프**: 이중검증·테스트 (무인이라 더 중요).
3. **게이트**: 비가역 액션만 멈춤 → 나머지는 논스톱.

### P59 [이미지] — 운영 원리
- 비주얼: 리더(메인) 1명이 여러 워커를 병렬 디스패치, 각자 격리된 작업대(worktree).

### P60 [설명] — 어떻게 길게 굴러가나
- 메인=리더 "빙의" → 일을 쪼개 병렬 디스패치.
- worktree 격리로 충돌 없이 동시 작업.

### P61 [이미지] — 수 시간 노하우
- 비주얼: 대시보드 — 컨텍스트 게이지·캐시 타이머·토큰 예산·loop 재개 표시.

### P62 [설명] — 자율주행 지속 팁 (Claude Max 관점)
- 컨텍스트 관리(요약·박제로 맥락 유지).
- 캐시 TTL(5분/1시간) 인지해 비용 절감.
- 토큰 예산으로 깊이 조절, loop로 끊겨도 자동 재개.

### P63 [이미지] — 🖥️ 라이브 데모
- 비주얼: "LIVE" 배지 + 진행 중인 sprint 대시보드 캡처.

### P64 [설명] — 데모 노트
- 박제된 design doc 하나로 sprint 실행 → 자율 진행 관찰.
- ⚠️ 현장 네트워크·한도 리스크 대비 **사전 녹화 백업** 재생 가능.
